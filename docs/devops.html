O que é DevOps e por que precisamos disso?

Antes de falar sobre a implementação, vale entender o problema que o DevOps resolve.

Imagine um time de desenvolvimento onde cada pessoa trabalha semanas no seu código, e só no final juntam tudo. Resultado: conflitos, bugs difíceis de rastrear e aquela clássica frase "na minha máquina funciona". Esse cenário era a realidade de praticamente todos os times antes do DevOps.

DevOps é a cultura e o conjunto de práticas que aproxima o desenvolvimento de software (Dev) da operação de infraestrutura (Ops). O objetivo central é entregar software com mais velocidade, mais confiança e menos atrito.

⚡
Integração Contínua (CI)

Todo código enviado é automaticamente compilado e testado. Erros aparecem em minutos, não em semanas. O time recebe feedback imediato sobre a qualidade do que foi escrito.

🚀
Entrega Contínua (CD)

Após passar pelos testes, o código está pronto para ser colocado em produção a qualquer momento. O processo de deploy deixa de ser um evento estressante e manual.

🔍
Qualidade automatizada

Ferramentas de análise estática verificam automaticamente cobertura de testes, bugs em potencial, vulnerabilidades de segurança e code smells — sem depender de revisão manual.

📦
Builds reproduzíveis

Com Docker e pipelines automatizadas, o ambiente de build é sempre o mesmo — eliminando a variação de "funciona na minha máquina mas não no servidor".

💡
A ideia central
Em vez de integrar código raramente e com dor, o DevOps propõe integrar frequentemente e com automação. Cada push dispara automaticamente uma sequência de verificações que garante que o código está saudável antes de chegar à main branch ou ao ambiente de produção.
2

O projeto Pooker

Uma plataforma de aluguel de filmes construída como monorepo com múltiplas tecnologias.

O Pooker é uma aplicação de aluguel de filmes estruturada como um monorepo — um único repositório Git que contém todos os serviços. A arquitetura é de microserviços: cada funcionalidade é um serviço independente com sua própria linguagem, banco de dados e responsabilidade.

Os 7 microserviços

Serviço Tecnologia Responsabilidade
api-gateway Java 17 · Spring Boot · Gradle Ponto de entrada único — roteia requisições para os serviços corretos
auth-service Java 17 · Spring Boot · Maven Autenticação e autorização — login, tokens JWT, controle de acesso
filmes-service Python 3.11 · FastAPI Catálogo de filmes — listagem, detalhes, busca
alugueis-service Python 3.11 · FastAPI Gestão de aluguéis — criar, listar, devolver
user-service Python 3.11 · FastAPI Cadastro e gerenciamento de usuários
payment-service Python 3.11 · FastAPI Processamento de pagamentos
frontend Angular 20 · TypeScript Interface web — SPA consumindo todos os serviços via API Gateway
⚠️
O desafio do monorepo com múltiplas tecnologias
Ter Java, Python e Angular no mesmo repositório significa que cada serviço precisa de ferramentas completamente diferentes para build, teste e análise. A pipeline precisaria ser inteligente o suficiente para saber o que fazer com cada um — e eficiente o suficiente para não recompilar tudo quando só um arquivo foi alterado.
3

Por que essas escolhas?

As decisões de arquitetura foram tomadas com base nas restrições reais do projeto — orçamento zero, máquina Windows local, prazo curto.

GitHub Actions — por que não Jenkins ou GitLab CI?

O GitHub Actions foi escolhido porque o código já estava hospedado no GitHub, eliminando a necessidade de configurar um servidor de CI separado ou pagar por uma ferramenta externa. O arquivo de configuração fica junto ao código, é versionado no Git e qualquer alteração na pipeline é rastreável como qualquer outro commit.

Runner self-hosted — por que não o runner em nuvem do GitHub?

O GitHub oferece runners na nuvem gratuitamente, mas com um limite de 2.000 minutos por mês para contas gratuitas. Com 7 serviços sendo testados, esse limite se esgotaria rapidamente. A solução foi usar um runner self-hosted: um programa instalado na própria máquina de desenvolvimento que conecta ao GitHub e executa os jobs localmente.

🖥️
Por que o runner ficou nativo no Windows (não em Docker)?
A ideia original era rodar o runner dentro de um container Docker. Mas o projeto já tem Docker Desktop rodando SonarQube e PostgreSQL — adicionar mais um container criaria dependências circulares e complicaria o acesso ao localhost. Instalar o runner nativo no Windows foi mais simples, mais direto e evitou problemas de rede entre containers.

SonarQube — por que analisar qualidade na pipeline?

Testes unitários verificam se o código funciona. Mas não dizem se ele é seguro, manutenível ou se está sendo testado o suficiente. O SonarQube faz análise estática — lê o código sem executá-lo — e detecta padrões problemáticos, vulnerabilidades de segurança e mede a cobertura de testes.

Escolhemos o SonarQube Community Edition por ser open source, gratuito e rodar em Docker com um único comando. Para um projeto acadêmico ou startup, é a opção mais acessível sem abrir mão de funcionalidades profissionais.

3 pipelines separadas — por que não uma só?

Uma única pipeline que roda tudo seria simples de configurar, mas extremamente ineficiente. Alterar um arquivo Python não deveria recompilar o código Java nem reinstalar as dependências do Angular. Com 3 pipelines independentes e filtros de path, cada serviço só executa quando seus arquivos mudam. 

# Só dispara quando arquivos do backend Java mudam
on:
  push:
    paths:
      - 'backend/api-gateway/**'
      - 'backend/auth-service/**'
4

Os componentes da esteira

Quatro peças que trabalham juntas para formar a infraestrutura de CI/CD.

⚡
GitHub Actions — Orquestrador
É o cérebro da esteira. Fica monitorando o repositório e, quando detecta um push ou PR nas branches main, master ou develop, dispara os jobs correspondentes. Toda a lógica fica em arquivos YAML dentro da pasta .github/workflows/.
🏃
Self-hosted Runner — Executor
Um processo rodando em segundo plano no Windows que fica em modo idle, aguardando ordens do GitHub Actions. Quando um job chega, ele faz o checkout do código, executa cada step em sequência e reporta o resultado de volta ao GitHub. Por rodar localmente, tem acesso direto ao SonarQube e ao Docker sem configurações de rede especiais.
🔍
SonarQube — Analisador de qualidade
Roda em um container Docker na porta 9000. Recebe os relatórios de cobertura gerados pelos testes (JaCoCo para Java, coverage.xml para Python, LCOV para Angular) e faz a análise estática do código. Ao final, emite um veredicto: Quality Gate passou ou Quality Gate falhou.
🐳
Docker — Empacotador
Ao final de uma pipeline bem-sucedida (com Quality Gate aprovado), cada serviço é empacotado em uma imagem Docker. Isso garante que o artefato gerado é sempre o mesmo independentemente do ambiente onde vai rodar — o mesmo container que passou nos testes é o que vai para produção.

As 3 pipelines em detalhe

Cada tecnologia tem seu próprio fluxo, adaptado às suas ferramentas nativas.

☕ Pipeline Java

Cobre o api-gateway (Gradle) e o auth-service (Maven). São dois jobs independentes dentro do mesmo arquivo YAML — rodam em paralelo, então não precisam esperar um pelo outro.

O build Java tem uma característica especial: separamos o build da execução dos testes. O primeiro passo compila o código inteiro para verificar erros de compilação. Se compilar, os testes rodam. Em seguida, o JaCoCo gera um relatório XML com a cobertura de cada linha e classe. Esse relatório é enviado ao SonarQube para análise.

🔧
Detalhe técnico: banco de dados nos testes
O auth-service usa PostgreSQL em produção, mas rodar um banco de dados no CI seria lento e complexo. A solução foi criar um arquivo src/test/resources/application.properties que substitui a configuração em tempo de teste — usando H2, um banco de dados em memória que não precisa ser instalado. Os testes rodam mais rápido e de forma totalmente isolada.

🐍 Pipeline Python

Cobre 4 serviços FastAPI de uma vez usando a estratégia de matrix do GitHub Actions. Em vez de copiar e colar o mesmo job 4 vezes, declaramos uma lista de serviços e o GitHub Actions executa o job uma vez para cada item — em paralelo. 

strategy:
  matrix:
    service: [alugueis-service, filmes-service, user-service, payment-service]
  fail-fast: false  # falha num serviço não cancela os outros

O lint com flake8 foi configurado para verificar apenas erros graves de sintaxe (E9, F821, F822, F823) — ignorando violações de estilo que existiam no código herdado. Assim a pipeline não falha por indentação ou linha longa, apenas por código que realmente não compila.

🅰️ Pipeline Angular

Roda o npm ci para instalar dependências de forma determinística (usando exatamente o que está no package-lock.json), executa o lint ESLint com continue-on-error para não bloquear por violações pré-existentes, e os testes com Karma + ChromeHeadless — um navegador Chrome sem interface gráfica, ideal para ambientes de CI.

A cobertura de código é gerada pelo Istanbul (integrado ao Angular) no formato LCOV e enviada ao SonarQube para análise. Por fim, o build de produção gera os arquivos otimizados em dist/.

6

A implementação em 3 fases

A esteira não foi implementada de uma vez. Construímos em fases para ter valor entregue a cada etapa.

Fase 1
Build e testes verdes
O objetivo da primeira fase era simples: os 3 pipelines executam sem erro. Isso significou configurar o runner, ajustar os YAMLs para PowerShell (o runner roda no Windows, não Linux), corrigir dependências quebradas do npm, e fazer os testes Java funcionarem sem precisar de banco de dados externo. Resultado: 3 pipelines verdes, runner idle aguardando jobs.
Fase 2
SonarQube conectado
Com os builds verdes, integramos o SonarQube. Cada serviço ganhou um arquivo sonar-project.properties com sua chave única. O token de autenticação e a URL do SonarQube foram guardados como secrets no GitHub — nunca no código. A variável SONAR_ENABLED=true foi criada no repositório para ligar/desligar a análise sem precisar alterar os YAMLs.
Fase 3
Quality Gate bloqueante
Na fase final, o Quality Gate passou a ser bloqueante: se um serviço não atender aos critérios mínimos de qualidade, o pipeline inteiro falha e o Docker build não acontece. Isso é controlado pelo parâmetro -Dsonar.qualitygate.wait=true, que faz o sonar-scanner aguardar o resultado antes de retornar o exit code para o GitHub Actions.
7

Qualidade de código com SonarQube

O que o SonarQube analisa e como o Quality Gate funciona na prática.

O SonarQube recebe dois tipos de entrada de cada serviço: o código-fonte e os relatórios de cobertura gerados pelos testes. Com esses dados, ele calcula métricas e compara com o Quality Gate configurado.

Como a cobertura chega ao SonarQube

☕
Java

O Gradle/Maven roda o JaCoCo após os testes e gera um jacocoTestReport.xml. O SonarQube lê esse arquivo para saber quais linhas foram executadas.

🐍
Python

O pytest-cov gera um coverage.xml no formato Cobertura. O SonarQube entende esse formato nativamente e calcula a cobertura por módulo.

🅰️
Angular

O Karma gera um arquivo LCOV em coverage/looker/lcov.info. O SonarQube usa esse arquivo para mapear cobertura de TypeScript.

O que acontece quando o Quality Gate falha

Quando o Quality Gate reprovada, a pipeline marca o step de SonarQube como vermelho e interrompe a execução — o Docker build não acontece. No GitHub, o check da pipeline aparece como falho, o que impede o merge de um Pull Request caso a branch protection esteja configurada.

🚦
O fluxo de decisão do Quality Gate
sonar-scanner envia análise
  ↓
SonarQube avalia condições
  ↓
cobertura ≥ 70% E bugs = 0 E vulnerabilidades = 0
  ↓
✓ PASSED → pipeline continua → Docker build
✗ FAILED → exit code 1 → pipeline falha → sem Docker build

Desafios reais durante a implementação

A esteira não saiu perfeita na primeira tentativa. Esses são os problemas reais que encontramos e como foram resolvidos.

🪟 PowerShell em vez de Bash

O runner roda no Windows, onde o shell padrão é PowerShell — não Bash. Os YAMLs do GitHub Actions usam Bash por padrão, então comandos como if [ -f arquivo ] ou cd pasta && comando falhavam com erro de sintaxe. A solução foi reescrever todos os scripts usando sintaxe PowerShell: if (Test-Path arquivo), working-directory: por step, e wrappers Windows (.\gradlew.bat, .\mvnw.cmd).

🔍 sonar-scanner não estava no PATH

O sonar-scanner não vem pré-instalado no Windows. A abordagem inicial foi baixar o ZIP em um step e adicionar o diretório ao $GITHUB_PATH para que o step seguinte pudesse invocar sonar-scanner diretamente. Isso não funcionou porque a variável de PATH só é relida no início de cada processo. A solução: combinar download e execução em um único step, usando o caminho completo do executável: 

# Em vez de dois steps separados, tudo em um:
run: |
  $scannerDir = "$env:USERPROFILE\.sonar\sonar-scanner-5.0.1.3006-windows"
  # ... download se não existir ...
  & "$scannerDir\bin\sonar-scanner.bat" "-Dsonar.qualitygate.wait=true"

💥 PowerShell partindo argumentos com ponto

Mesmo com o caminho completo, a chamada sonar-scanner.bat -Dsonar.qualitygate.wait=true falhava. O PowerShell interpretava -D como flag e sonar.qualitygate.wait=true como argumento solto. A correção foi colocar aspas no argumento inteiro: "-Dsonar.qualitygate.wait=true".

📦 package-lock.json fora de sincronia

O npm ci falhou porque o package-lock.json estava desatualizado em relação ao package.json — pacotes do Angular ESLint tinham sido adicionados mas o lock não foi regenerado. A solução foi rodar npm install localmente para atualizar o lock file e commitar o resultado.

📁 Caminho do LCOV errado

O SonarQube apontava para coverage/lcov.info mas o Angular gera o arquivo em coverage/nome-do-projeto/lcov.info. O projeto se chama looker no angular.json, então o caminho correto era coverage/looker/lcov.info.

✅
Aprendizado geral
A maior parte dos problemas veio da diferença entre o ambiente esperado (Linux/Bash, que é o padrão dos tutoriais de CI) e o ambiente real (Windows/PowerShell). Cada erro ensinou algo sobre como o runner interpreta scripts — e a esteira final é mais robusta por ter passado por esses problemas reais.

Resultado final

O que foi entregue e o que isso representa para o projeto.

✅
3 pipelines funcionando

Java, Python e Angular têm pipelines independentes que disparam automaticamente em cada push e produzem relatórios de teste e cobertura.

🔍
7 projetos no SonarQube

Cada microserviço tem seu próprio projeto no dashboard do SonarQube com métricas de cobertura, bugs, vulnerabilidades e code smells.

🚦
Quality Gate bloqueante

Código que não atende aos critérios de qualidade mínimos não passa pela pipeline. O Docker build só acontece após aprovação do Quality Gate.

💰
Custo zero

Toda a infraestrutura usa ferramentas open source — GitHub Actions self-hosted, SonarQube Community, Docker Desktop — sem custo de cloud.

O que a esteira garante em cada push

1
O código compila
Erros de sintaxe e compilação são detectados imediatamente, antes de chegar a qualquer revisão humana.
2
Os testes passam
Toda a suíte de testes unitários roda automaticamente. Uma regressão é detectada antes de chegar à branch principal.
3
A qualidade está dentro do padrão
O SonarQube garante que cobertura mínima foi atingida e que nenhum bug ou vulnerabilidade crítica foi introduzida.
4
A imagem Docker está pronta
Apenas código que passou em todas as etapas anteriores gera uma imagem Docker — o artefato pronto para deploy em qualquer ambiente.
🎯
Em resumo
O que foi construído aqui é exatamente o que times profissionais usam em empresas de tecnologia. A diferença de escala existe — grandes empresas têm mais serviços, mais stages de deploy, ambientes de staging e produção separados — mas o princípio é o mesmo: automatizar verificações, medir qualidade continuamente e garantir que só código saudável avança no processo.
